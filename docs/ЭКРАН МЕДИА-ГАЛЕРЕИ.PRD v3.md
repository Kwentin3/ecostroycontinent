ЧЕРНОВИК PRD v3
ЭКРАН МЕДИА-ГАЛЕРЕИ.PRD v3

Статус документа

Предварительный PRD-draft v3.
Документ фиксирует продуктовую рамку и жёсткие правила для нового единого экрана медиа-галереи в админке.
Это feature-level PRD, не implementation spec.

1. Контекст и причина эпика

В проекте уже существует media-domain как реальная доменная поверхность:
MediaAsset — отдельная first-class сущность;
Gallery — отдельная lightweight grouping сущность;
metadata truth живёт в SQL/content revisions;
binary truth живёт в object storage;
public delivery идёт через delivery/CDN path, а не через raw editorial truth.

При этом текущий media screen остаётся form-first:
оператор работает не с визуальной библиотекой, а с generic entity form;
визуальный browse-flow слабый;
usage visibility слабая;
после загрузки asset попадает в неудобный draft/reuse сценарий;
на media screen видны generic SEO-looking поля, которые не подтверждены persistence contract `media_asset`.

Это не косметическая проблема, а contract/UI leakage.

Цель эпика — заменить это на одно понятное рабочее место SEO/content-оператора:
визуальная галерея,
быстрый выбор,
понятные сигналы готовности,
честная правка metadata,
встроенный image-edit flow,
понятный reuse.

2. Что не переоткрываем

Этот эпик не переоткрывает:
Admin Console как write-side content operations tool, а не page builder и не enterprise DAM;
разделение `MediaAsset` и `Gallery` как разных сущностей;
разделение metadata truth и binary truth;
public delivery через CDN/public URL, а не через editorial source of truth;
phase 1 как narrow launch-core, без разрастания в heavy DAM/workflow platform;
right rail как support/status/history only, а не main metadata workspace.

3. Продуктовая идея

В интерфейсе вводится один канонический экран-рабочее место: экран медиа-галереи.

Это не означает, что `MediaAsset` и `Gallery` становятся одной сущностью.
Это означает, что оператору больше не нужно держать в голове два разрозненных экрана с пересекающимся смыслом.

Экран медиа-галереи становится основным местом для:
загрузки нового изображения;
поиска существующего ассета;
визуального просмотра библиотеки;
быстрой оценки готовности;
редактирования metadata;
image-edit режима;
создания производного варианта;
понимания, где asset используется;
повторного использования asset в service/case/page/gallery связях.

4. Основная роль

Primary role:
SEO manager / content operator.

Secondary roles:
superadmin;
owner/reviewer.

Базовый рабочий сценарий роли:
пользователь открывает экран как рабочее место смены;
за 10–30 секунд понимает, что новое, что проблемное, что без alt, что сломано, что не используется;
либо дорабатывает существующий asset,
либо загружает новый и доводит его до рабочего состояния.

5. Проблема, которую решаем

Текущие боли:
экран ощущается как generic form, а не media library;
нет нативного gallery-first browse-flow;
нет удобных операторских фильтров;
после upload asset не проходит естественный finish-first сценарий;
неочевидно, где asset уже используется;
нет честного, встроенного image-edit flow;
UI показывает поля, которые не являются подтверждённым persisted contract для `media_asset`.

6. Цель фичи

Сделать единый gallery-first экран, на котором оператор может:
быстро найти existing asset;
быстро понять его readiness/problems;
загрузить новый asset;
сразу довести новый asset до рабочего состояния;
заполнить честные metadata fields;
увидеть usage;
создать derived variant при необходимости;
вернуться в галерею без потери контекста.

7. Non-goals

В рамках этого эпика не делаем:
enterprise DAM suite;
heavy workflow engine;
advanced portfolio/filtering platform;
SEO dashboard;
broad analytics surface;
Photoshop-class editor;
автономную AI-обработку и публикацию;
массовую платформу bulk-операций;
широкую future-архитектуру beyond phase 1.

8. Навигационная рамка

На уровне рабочего места остаётся один канонический экран: медиа-галерея.

Навигационное правило:
левый sidebar остаётся глобальной навигацией и не превращается в дерево медиа или вложенную файловую систему;
если `Gallery` как grouping entity сохраняется как отдельная сущность, управление её составом и связями допускается из того же рабочего места через filters, related panels или nested surface, но не через второй дублирующий экран “правды”;
top breadcrumb/depth bar остаётся каноническим nested-surface паттерном, если понадобится углубление в состав группы или usage details.

Итог:
одно рабочее место;
без второго конкурирующего экрана по смыслу;
без стирания доменной разницы между asset и gallery.

9. Честный metadata contract для V1

Для `media_asset` в V1 считаются честно поддержанными только следующие редактируемые поля:
title;
alt;
caption;
sourceNote;
ownershipNote.

Явное разведение полей:
title — редактируемое человекочитаемое имя ассета;
originalFilename — системное поле, пришедшее из upload;
originalFilename не редактируется оператором.

Текущие generic SEO fields shared-form machinery не считаются частью persisted contract `media_asset` и не должны показываться как обычные рабочие поля на экране медиа-галереи, пока не будет отдельного подтверждённого contract decision.

Жёсткое правило V1:
если поле не подтверждено persistence contract `media_asset`, оно не должно показываться как нормальное редактируемое поле на media surface.

Отдельная оговорка:
у `gallery` SEO contract реален и persisted.
Это не даёт права автоматически переносить тот же SEO block на `media_asset`.

10. Целевая UX-модель

Один экран состоит из трёх режимов.

Режим 1. Обзор галереи.
Пользователь видит:
toolbar сверху;
grid карточек media assets;
при выборе карточки — right inspector panel.

Режим 2. Выбранный asset.
Пользователь остаётся на том же экране и видит:
краткий инспектор;
warnings;
usage visibility;
главное действие “Редактировать”.

Режим 3. Большой edit overlay.
По действию “Редактировать” или сразу после upload открывается большой overlay/modal поверх галереи, где пользователь:
видит крупное превью;
редактирует metadata;
при необходимости переходит во вкладку/режим “Изображение”;
сохраняет изменения;
возвращается в галерею на тот же asset.

11. Верхняя панель

Верхняя панель должна содержать:
название экрана;
поиск;
сортировку;
быстрые фильтры;
primary CTA “Загрузить”.

Drag-and-drop загрузка на галерейную область желательна и должна поддерживаться, если не ломает V1-сложность.
При этом primary и гарантированный путь V1 — кнопка “Загрузить”.
Отсутствие D&D не является критическим блокером релиза, но D&D желательно закладывать сразу.

12. Галерейная область

Основная область — это grid карточек, а не table/list-first surface.

Каждая карточка показывает только load-bearing сигналы:
preview;
название;
status;
badge alt / !alt;
usage signal;
если нужно — compact type/date/size.

Карточка не должна быть перегружена длинными полями или сложной формой.

13. Обязательные сигналы на карточке

На карточке V1 должны быть видны:
есть ли preview;
есть ли alt;
статус draft/review/published или локальный equivalent;
используется / не используется;
broken state, если metadata существует, но preview/public/admin binary path не отрабатывает.

Broken asset signal обязателен.
Для оператора важно не только “есть alt или нет”, но и “вообще доступен ли файл”.

14. Поиск, сортировка и фильтры V1

Поиск V1:
по title;
по original filename;
по alt;
по caption;
по sourceNote / ownershipNote, если это не создаёт непропорциональной дороговизны.

Сортировка V1:
по дате;
по имени;
по status/readiness.

Быстрые фильтры V1:
все;
загружено недавно;
загружено мной;
без alt;
используется;
не используется;
draft;
review;
published.

Фильтры “используется в услугах” и “используется в кейсах” не входят в обязательную V1 по умолчанию.
Они допустимы в V1.1/V2 или в V1, если оказываются почти бесплатными по текущему contract/query layer.

15. Правая панель-инспектор

Правая панель нужна как быстрый контекстный слой, но не как главный редактор.

Правая панель показывает:
увеличенное превью;
title / original filename;
mime / size;
status;
alt presence warning;
caption preview;
source / ownership hints;
used / unused;
last edited / updated;
broken / not broken;
quick warnings.

В правой панели должна быть одна главная action-кнопка:
“Редактировать”.

Кнопки “Открыть редактор изображения” и “Создать вариант” не живут отдельными равноправными действиями в inspector.
Они переносятся внутрь большого edit overlay.

16. Большой edit overlay

Большой overlay — это нормальное рабочее место редактирования внутри того же экрана.

Он содержит:
крупное превью;
режим “Метаданные”;
режим “Изображение”;
save/cancel;
если нужно — действие “Создать вариант”.

Overlay не должен ощущаться как переход на отдельную страницу сущности.

Правило для edit existing asset:
после save пользователь возвращается в галерею на тот же asset;
selection state сохраняется, если нет специальных причин его сбросить.

Дополнительное правило для create/save нового asset:
после save нового asset UI обязан обеспечить find-me-after-save behavior.
Новый asset становится selected, а его позиция в grid должна быть явно обнаружима пользователем независимо от текущей сортировки.

Допустимые реализации:
временное поднятие нового asset наверх;
автоскролл к его реальной позиции;
визуальная подсветка нового asset в grid.

PRD фиксирует не конкретную реализацию, а обязательный результат:
пользователь не должен “потерять” только что созданный asset после save.

17. Image editing flow

Редактирование изображения не делается в right rail и не делается в нижней панели.
Для image editing нужен полноценный большой слой, потому что crop и визуальная правка требуют пространства.

В V1 допустимы:
crop;
rotate;
flip;
basic correction/finetune;
resize, если не создаёт непропорциональный scope;
создание derived variant.

Кандидат для image-edit layer:
react-filerobot-image-editor.

18. Политика overwrite / variant

Это жёсткое contract rule V1.

Правило:
overwrite допустим только для draft assets;
overwrite запрещён для published assets;
для published assets основной безопасный путь — создать derived variant с явным последующим выбором, где именно он будет использован.

Причина:
молчаливый overwrite published asset может сломать live case/service/page.

19. Upload flow V1

Жёсткий V1-сценарий:
пользователь нажимает “Загрузить” или бросает файл drag-and-drop;
после выбора файла открывается большой edit overlay;
пользователь видит превью;
при необходимости делает image edit;
заполняет честные metadata fields;
сохраняет;
новый asset появляется в галерее уже как осмысленный объект.

Принцип:
после upload пользователь не должен получать “сырой файл, который потом где-то ещё нужно оформить”.

Ограничение V1:
single-item finish-first workflow — основной канон.
Простой multi-upload допустим только если он не превращается в отдельную queue-orchestration систему.
Heavy batch queue не входит в V1.

Upload failure handling V1:
если upload или save завершается ошибкой, ошибка показывается явно в том же edit overlay;
оператор не теряет уже введённые metadata;
оператор может повторить попытку или отменить действие без молчаливой потери данных.

Примеры ошибок:
неподдерживаемый формат;
слишком большой файл;
storage/upload failure;
timeout;
несогласованность между binary upload и metadata save.

Жёсткое правило:
если binary не был успешно принят и asset не был корректно сохранён, UI не должен тихо показывать его как нормальный рабочий объект в галерее.

20. Selection state и поведение экрана

Selection rules:
при клике по карточке asset становится selected;
selected state должен переживать открытие/закрытие inspector;
selected state должен по возможности сохраняться после save existing asset;
после upload и save новый asset становится selected;
если применён фильтр, скрывающий выбранный asset, selection корректно сбрасывается на none;
Escape закрывает overlay;
повторный Escape закрывает inspector, если overlay уже закрыт;
delete/archive выбранного asset должен корректно переводить selection либо на соседний элемент, либо в none.

21. Delete / archive lifecycle

Hard delete не является default lifecycle path для content/media.

Правило V1:
для media по умолчанию используется archive / withdraw / quarantine posture, а не hard delete.

Жёсткое безопасное правило V1:
любое удаление или архивирование блокируется, если у asset есть хотя бы одна usage reference — draft или published.

Это намеренно более грубое правило первой версии:
оно проще, безопаснее и убирает серые зоны.

Следствие:
destructive delete при любом usage недоступен;
archive/withdraw при любом usage тоже недоступен без отдельного специального сценария;
оператор должен получать понятное объяснение, почему действие недоступно и какие безопасные действия доступны вместо него.

22. Usage visibility

Экран должен отвечать на базовые вопросы:
используется ли asset вообще;
сколько у него usage references;
безопасно ли его менять;
можно ли его переиспользовать.

Минимум V1:
used / unused;
usage count;
возможность увидеть usage summary.

Детализация по типам сущностей допускается как V2 или как V1.1, если текущая модель это даёт почти бесплатно.

23. Gallery vs Media

Интерфейс становится единым рабочим местом.
Но доменно:
MediaAsset — отдельный asset;
Gallery — lightweight ordered grouping.

Следовательно:
нельзя превратить всё в одну размытую сущность “любая медиа штука”;
нельзя стирать разницу между asset library и grouping semantics;
нельзя случайно притянуть gallery SEO contract на media asset.

24. Accessibility и keyboard behavior

Это обязательное UX-требование V1.

Общие правила:
фокус должен быть видимым;
selection state должен быть понятен визуально;
карточки должны быть достижимы с клавиатуры;
Escape работает предсказуемо по слоям.

Правила навигации по grid:
Tab позволяет войти в галерейную область и дойти до активной карточки;
стрелки клавиатуры перемещают фокус между карточками в grid;
Enter или Space выбирают карточку;
если inspector уже открыт, Enter из выбранной карточки должен вести в предсказуемый selection/edit path, без двусмысленности.

PRD фиксирует обязательность keyboard navigation по grid как часть V1 UX/accessibility behavior.

25. Минимальные требования V1

Functional:
grid cards вместо list-first/table-first;
поиск;
сортировка;
быстрые фильтры V1;
выбор карточки;
inspector panel;
upload;
edit overlay;
редактирование честных metadata fields;
image edit mode;
usage visibility;
alt/broken/status signals прямо в галерее.

UX:
не заставлять пользователя постоянно уходить на отдельную страницу сущности;
не перегружать right rail длинной формой;
не обещать поля, которых нет в contract;
сохранять ощущение одного рабочего места;
делать новый asset рабочим сразу после upload;
не терять новый asset после save;
не терять введённые metadata при upload/save error.

Architecture-facing:
не ломать media/storage/publish boundaries;
не тащить heavy DAM scope;
не вводить silent overwrite published assets;
не вводить contract-lie fields.

26. Риски

Смешать `MediaAsset` и `Gallery` в одну кашу.
Оставить на экране поля, которых нет в contract.
Случайно разрешить overwrite published assets.
Перегрузить inspector panel.
Раздуть scope V1 до mini-DAM.
Уйти в batch/queue раньше времени.
Не закрыть broken asset state.
Сделать красивую галерею без реальной operator value в reuse/usage.
Сделать поведение после create/save непредсказуемым.
Оставить error handling на усмотрение реализации.

27. Что уже можно считать согласованным

Один канонический экран-рабочее место.
Gallery-first browse model.
Right panel как inspector, не main editor.
Большой overlay для нормальной редактуры.
Image editing в overlay.
Честный узкий metadata contract V1 для media.
Title и original filename — разные поля.
Запрет overwrite published assets.
Alt/broken/status/use signals прямо на карточке.
Упрощённые V1 filters.
Single-item finish-first upload workflow.
Find-me-after-save behavior для нового asset.
Блокировка удаления/архивации при любом usage.
Явная keyboard navigation по grid.

28. Что ещё требует явного подтверждения

Нужен ли D&D уже в строгом V1 или можно выпустить сразу после.
Какая именно usage summary показывается в inspector.
Как именно визуализируется archive/withdraw lifecycle.
Нужен ли simple multi-upload в первой поставке или только single-item.

29. Короткий итог

Экран медиа-галереи должен стать не страницей сущности и не файловым списком, а визуальным рабочим местом SEO/content-оператора.

Его формула такая:
галерея — для обзора и выбора;
инспектор справа — для контекста и предупреждений;
большой overlay — для нормальной правки;
честный metadata contract — без UI-лжи;
published assets — без overwrite;
reuse и usage — обязательная часть ценности, а не nice-to-have.

Если захочешь, следующим сообщением я могу сделать из этого ещё более плотную версию — **owner-ready one-page summary без повторов**, чтобы по ней быстро проходиться глазами и принимать решения.

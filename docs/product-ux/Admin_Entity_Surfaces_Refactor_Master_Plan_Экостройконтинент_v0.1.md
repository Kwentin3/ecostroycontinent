## 1. Короткий вывод

Следующий этап нужно вести не как набор локальных косметических правок по отдельным экранам, а как единый refactor program для административных карточек сущностей.

Проблема носит системный характер:
- экраны перегружены крупным UI chrome;
- повторяются одни и те же лейблы и секционные паттерны;
- инженерная терминология слишком заметна в повседневной работе;
- полезная форма и доменная задача уходят ниже первого экрана;
- SEO-специалист получает тяжёлую админку вместо лёгкого рабочего инструмента.

Главная цель рефакторинга:

`Превратить административные экраны сущностей в компактные, спокойные и редактирующие-first рабочие места, где основной доменный сценарий понятен с первого взгляда, помещается в первый экран desktop и не заглушается инженерной обвязкой.`

## 2. Почему это отдельный мастер-план

Текущее состояние уже нельзя честно описывать как проблему только карточки `Техника`.

По накопленному контексту видно повторяющийся паттерн:
- высокий hero до начала реальной формы;
- крупные action rows и pills;
- перегруженные правые панели;
- смешение редактирования с диагностикой;
- терминология, полезная разработчику или супер-админу, но лишняя для SEO-специалиста;
- слабая информационная плотность на первом экране.

Если чинить это локально по одной сущности, получится серия частных компромиссов.

Если чинить это как общую систему, можно:
- унифицировать shell;
- уменьшить scale базовых UI primitives;
- убрать дубли разом;
- сделать один понятный стандарт для всех entity cards.

## 3. Северная звезда рефакторинга

Правильный admin screen для SEO-специалиста должен ощущаться так:
- открыл карточку и сразу понял, что здесь главное;
- начал редактирование без предварительного прокручивания через шапки и статусы;
- видит только те действия, которые реально нужны в текущем рабочем цикле;
- не отвлекается на служебные и инженерные сущности;
- не расшифровывает терминологию;
- не сталкивается с повторяющимися блоками, лейблами и секциями.

Ключевая формула:

`less chrome, more work`

## 4. Основные проблемы, которые уже подтверждены

### 4.1. Дублирование и терминологический шум

Самый явный симптом:
- `Поисковая оптимизация / данные` повторяется многократно на одном и том же экране.

Это создаёт сразу несколько проблем:
- визуальный шум;
- ложное ощущение сложности;
- слабая структура страницы;
- дублирование смысла вместо ясной навигации по секциям.

Отдельно надо провести аудит на:
- повторяющиеся eyebrow-лейблы;
- повторяющиеся названия секций;
- повторяющиеся действия;
- дублирование статусов и summary-информации;
- повторяющиеся объяснительные тексты.

### 4.2. Инженерный слой слишком близко к основному редактору

Такие элементы, как:
- `Реестр доказательств`;
- `Центр очистки`;
- `legacy`;
- доказательный слой;
- системные объяснения про truth/domains

не должны быть главным слоем повседневного интерфейса SEO-специалиста.

Они либо нужны редко, либо нужны другому типу пользователя, либо должны жить:
- глубже;
- тише;
- за secondary affordance;
- в отдельной диагностической поверхности.

### 4.3. Экранный вес не соответствует объёму данных

Во многих местах данных немного, но интерфейс выглядит тяжёлым из-за:
- крупных hero-блоков;
- больших вертикальных отступов;
- oversized pills и кнопок;
- постоянных сайдбаров;
- повторяющихся технических карточек.

В результате:
- растёт скролл;
- снижается концентрация;
- оператор хуже чувствует структуру формы;
- каждое редактирование ощущается “тяжёлой процедурой”.

### 4.4. Приоритеты действий размыты

На экранах спорят между собой:
- primary save / submit actions;
- история;
- служебные действия;
- опасные или редкие lifecycle-действия;
- legacy-переходы.

Для SEO-специалиста это ухудшает понятность:
- что делать сейчас;
- что является главным действием;
- что относится к редкому обслуживанию, а не к ежедневной работе.

## 5. Продуктовые принципы, которые должны управлять рефакторингом

### 5.1. SEO-specialist first

Интерфейс проектируется от повседневной работы SEO-специалиста, а не от полноты внутренних системных возможностей.

Вопрос по умолчанию должен звучать так:

`Помогает ли этот элемент быстрее и спокойнее выполнить рабочую задачу на карточке сущности?`

Если ответ нет, элемент:
- убирается;
- упрощается;
- переносится;
- сворачивается;
- становится вторичным.

### 5.2. One-screen-first editing

На desktop основной edit-flow должен начинаться в первом экране, а в идеале большая часть часто используемых полей должна быть доступна без заметного initial scroll.

Это не жёсткая математическая догма, но это правильное направление.

### 5.3. One concept -> one place -> one label

Если смысл уже обозначен, его не нужно повторять шесть раз.

Для каждого типа информации должен быть один основной способ показа:
- один лейбл;
- одно место в иерархии;
- одна доменная формулировка.

### 5.4. Primary editing surface отдельно от secondary diagnostics

Редактирование должно быть первым слоем.

Диагностика, форензика, служебные реестры и технические подтверждения:
- не должны конкурировать за внимание;
- не должны занимать половину первого экрана;
- не должны выглядеть равноправной частью базовой SEO-формы.

### 5.5. Compact by default

Базовое состояние карточки должно быть:
- компактным;
- визуально тихим;
- легко сканируемым;
- без ощущения dashboard inflation.

### 5.6. Progressive disclosure instead of permanent overload

Редкие, опасные, служебные и инженерные функции должны открываться по запросу, а не жить постоянно в основном потоке.

## 6. Целевая модель административной карточки сущности

Каждая entity card должна быть устроена примерно так.

### 6.1. Верхняя зона

Сверху остаётся только действительно нужное:
- название сущности;
- краткий статус;
- 1-2 главных действия;
- компактный secondary actions menu.

Из верхней зоны нужно убрать:
- избыточный hero;
- повторяющиеся summary-блоки;
- длинные объяснительные тексты;
- дубли действий.

### 6.2. Основная форма

В центре должен жить основной доменный сценарий:
- заголовок;
- ключевые SEO/content поля;
- связи и доменные данные;
- короткие, понятные секции без повторяющихся префиксов.

Каждая секция должна отвечать на понятный вопрос, например:
- что это;
- как называется;
- как ищется;
- что связано;
- что увидит пользователь.

### 6.3. Правая панель

Правая панель не должна быть permanent heavyweight rail по умолчанию.

Допустимые варианты:
- collapseable rail;
- secondary tab;
- drawer;
- блоки ниже формы;
- режим показа только при необходимости.

В правой панели должны оставаться только действительно полезные соседние задачи:
- краткая проверка готовности;
- контекстный preview;
- очень короткий служебный summary.

### 6.4. Инженерные и служебные функции

Такие вещи, как forensic, cleanup, evidence, legacy-операции и расширенная диагностика, должны быть вынесены в:
- `Дополнительно`;
- `Служебное`;
- `Диагностика`;
- отдельный superadmin layer.

Они не должны определять характер основного интерфейса.

## 7. Приоритеты рефакторинга

### P0. Убрать системное дублирование и шум

Это самый срочный продуктовый слой.

Нужно:
- провести cross-screen audit всех повторяющихся eyebrow-лейблов и секционных заголовков;
- убрать дубли `Поисковая оптимизация / данные`;
- схлопнуть повторяющиеся summary-блоки;
- убрать повторяющиеся действия вроде множественных входов в `История`;
- вычистить повторяющиеся инженерные объяснения.

Почему это P0:
- даёт быстрый выигрыш почти на всех экранах;
- снижает визуальную усталость;
- делает интерфейс понятнее без сложной архитектурной переделки.

### P1. Сжать shell и вернуть первый экран пользователю

Нужно:
- уменьшить высоту верхней части экрана;
- уменьшить scale кнопок, pills и карточек;
- пересобрать spacing system;
- сделать более плотный вертикальный ритм;
- довести экраны до one-screen-first posture.

Почему это P1:
- сейчас именно shell создаёт ощущение громоздкости;
- проблема кросс-экранная;
- выигрывает вся админка сразу.

### P1. Развести primary editing и secondary diagnostics

Нужно:
- пересмотреть роль right rail;
- вынести служебные блоки в secondary layer;
- оставить в основном потоке только то, что помогает редактированию.

Почему это P1:
- это главный structural источник ощущения “инженерной админки”.

### P1. Очистить язык интерфейса

Нужно:
- переписать названия блоков и действий на язык предметного специалиста;
- убрать без необходимости слова вроде `legacy`, `реестр`, `доказательства`, `truth`, если они не помогают задаче;
- сократить объяснительный copy до короткого рабочего уровня.

Почему это P1:
- терминология напрямую влияет на понятность и спокойствие интерфейса.

### P2. Упорядочить action architecture

Нужно:
- выделить один primary action pattern;
- все редкие и опасные операции убрать в menu;
- lifecycle и maintenance-actions сделать явно вторичными;
- убрать конкуренцию действий на первом экране.

### P2. Дошлифовать сущностные различия после общей чистки

Только после общей зачистки стоит тонко настраивать:
- `Техника`;
- `Услуги`;
- `Кейсы`;
- `Страницы`;
- `Медиа`.

Иначе есть риск подгонять частные экраны под старую тяжёлую систему.

## 8. Программа рефакторинга по волнам

### Wave 0. Cross-screen audit and inventory

Цель:
- подтвердить масштаб проблемы на всех ключевых entity screens;
- собрать список дублей, лишних блоков и инженерных элементов.

Что делаем:
- пройти `Техника`, `Услуги`, `Кейсы`, `Страницы`, `Медиа`;
- для каждого экрана зафиксировать:
  - высоту до начала основной формы;
  - обязательность скролла;
  - количество повторяющихся лейблов;
  - состав правой панели;
  - количество верхних действий;
  - инженерные термины на первом экране.

Артефакт:
- единая audit matrix по всем сущностям.

### Wave 1. Shared shell compression

Цель:
- уменьшить общий экранный вес админки без изменения доменной модели.

Что делаем:
- режем hero inflation;
- уплотняем spacing;
- уменьшаем action rows;
- делаем более компактные статусы и summary;
- выравниваем header pattern между сущностями.

### Wave 2. Duplicate collapse and copy cleanup

Цель:
- убрать дублирование и привести интерфейс к ясному доменному языку.

Что делаем:
- удаляем повторяющиеся eyebrow-лейблы;
- объединяем одинаковые секции;
- переписываем copy в language of task;
- вычищаем инженерную лексику из основного operator flow.

### Wave 3. Diagnostic demotion

Цель:
- перестать делать из обычной карточки сущности инженерную станцию управления.

Что делаем:
- переносим diagnostics в secondary surfaces;
- правую панель делаем контекстной или сворачиваемой;
- service/forensic/legacy-affordances выводим из primary layer.

### Wave 4. Entity-specific polishing

Цель:
- после общей нормализации довести каждую сущность до хорошего daily-use состояния.

Что делаем:
- адресуем специфические поля и сценарии для каждой карточки;
- проверяем, что общий shell не ломает особые доменные сценарии;
- поджимаем последние частные перегрузы.

### Wave 5. Operator validation

Цель:
- убедиться, что после рефакторинга интерфейс стал реально спокойнее и быстрее.

Что делаем:
- повторяем live product test;
- проходим типовые SEO-сценарии;
- проверяем, что реальная работа не требует лишней прокрутки и расшифровки UI.

## 9. Отдельный трек: аудит на дублирование

Этот трек должен идти с высоким приоритетом и отдельной дисциплиной.

Нужно собрать inventory по следующим типам дублей:
- одинаковые секционные eyebrow-лейблы;
- одинаковые названия смысловых блоков;
- повторяющиеся summary-блоки в hero и ниже;
- повторяющиеся действия в header, toolbar и footer;
- повторяющиеся технические пояснения;
- повторяющиеся статусы, если один и тот же статус уже виден выше.

Для каждого дубля нужно принять одно из решений:
- удалить;
- объединить;
- спрятать за disclosure;
- оставить только в одном каноническом месте.

Отдельное правило:

`Если блок не добавляет новый смысл, он не должен жить вторым экземпляром на том же экране.`

## 10. Отдельный трек: очистка интерфейса от инженерной обвязки

Нужно разделить два мира:

1. Повседневная работа SEO-специалиста.
2. Служебная, диагностическая и супер-админская работа.

Для первого мира на экране должны остаться:
- понятные доменные поля;
- ясные связи;
- статус готовности;
- сохранение, проверка, публикационный next step.

Для второго мира можно оставить:
- диагностику;
- forensic/evidence;
- cleanup;
- legacy-операции;
- расширенные системные объяснения.

Но этот второй мир не должен по умолчанию формировать характер экрана.

## 11. Что не надо делать в этом рефакторинге

Чтобы не расползтись, план должен держать границы.

Не надо:
- заново открывать доменную модель сущностей;
- превращать админку в generic CMS builder;
- смешивать этот UX-рефакторинг с полным redesign публичного сайта;
- добавлять новые сложные control-panels вместо удаления старых;
- лечить перегрузку добавлением новых подсказок поверх уже перегруженного UI;
- оставлять старый шум и просто делать его “чуть красивее”.

## 12. Критерии успеха

Рефакторинг можно считать успешным, если после него:
- первый экран каждой ключевой карточки явно принадлежит редактированию, а не chrome;
- оператор видит меньше дублей и меньше шумовых лейблов;
- инженерная лексика почти исчезает из основного потока;
- right rail перестаёт доминировать;
- primary action очевиден;
- ежедневный SEO-сценарий стал короче, спокойнее и понятнее;
- как минимум основные карточки ощущаются как рабочие инструменты, а не тяжёлые dashboard screens.

## 13. Рекомендуемый порядок исполнения

1. Сначала провести быстрый cross-screen audit и inventory дублей.
2. Затем принять shared rules для shell, sidebar, actions и copy.
3. После этого сделать общую волну уплотнения и очистки.
4. Только потом проходить отдельные сущности с их частными нюансами.
5. В конце подтвердить результат повторным live product test.

## 14. Следующий практический шаг

Следующее разумное действие после этого мастер-плана:

`Собрать короткую cross-screen audit matrix по экранам Техника / Услуги / Кейсы / Страницы / Медиа, чтобы из гипотезы о системной проблеме перейти к управляемой программе изменений с конкретным списком дублей и лишних инженерных блоков.`

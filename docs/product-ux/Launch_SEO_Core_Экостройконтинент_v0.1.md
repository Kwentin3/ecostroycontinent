# Launch SEO Core Spec v0.1

Проект: «Экостройконтинент»  
Статус: execution spec for phase 1 launch core  
Основание: [PRD v0.3.2](./PRD_Экостройконтинент_v0.3.1.md), [Content Contract v0.2](./Content_Contract_Экостройконтинент_v0.2.md)

## 1. Purpose of This Spec

Этот документ переводит текущий канон в конкретное launch-решение для SEO-ядра и стартовой структуры сайта.

Он не заменяет PRD, не пытается быть полным keyword-research handbook и не расширяет phase 1 beyond current canon. Его задача проста: зафиксировать, что именно мы запускаем сначала, чтобы следующим шагом собрать content inventory и editorial backlog.

## 2. Recommended Primary Region Cluster

### Recommended primary cluster

`Сочи / Большой Сочи`

Практическая логика выбора:

- у компании публично виден якорь в Сочи;
- в локальной выдаче заметен живой и конкурентный спрос на домостроение и монолитные работы именно по запросам с `Сочи`;
- для phase 1 это даёт самый чистый региональный фокус без расползания в multi-region SEO.

### Reserve option

`Адлер / Сириус`

Использовать только если реальный стартовый пул кейсов, объектов и входящих запросов почти полностью сосредоточен именно там.

### Operating rule

На запуске используется один primary region cluster. В titles, H1, proof blocks, `Contacts` и `Global Settings` не должно быть расплывчатого "работаем по всему югу России", если это не подтверждается реальным операционным контуром.

## 3. Launch Money-Page Core

Рекомендуемое ядро: `6` service pages. Это достаточно узко для phase 1, но уже покрывает основной коммерческий контур.

### 3.1 Service 01

- Working title: `Строительство домов под ключ`
- Page intent: high-intent запросы на полный цикл строительства частного дома.
- Candidate slug: `stroitelstvo-domov-pod-klyuch`
- Working H1: `Строительство домов под ключ в Сочи`
- Role in launch-core: anchor money page для частного домостроения; основной маршрут с home page.
- Primary CTA: `Получить консультацию и предварительный расчёт`
- Business Owner review: `Yes`
- Proof assets needed: минимум 2 case refs по частным домам, фото этапов, этапность работ, понятный scope "что входит / что не входит", FAQ по срокам и стоимости, фактура по договору и процессу.

### 3.2 Service 02

- Working title: `Монолитные работы`
- Page intent: запросы на монолит, бетонные конструкции, фундамент, подпорные стены, каркас.
- Candidate slug: `monolitnye-raboty`
- Working H1: `Монолитные работы в Сочи`
- Role in launch-core: технологическая money page с сильным локальным соответствием рельефу, нагрузкам и конструктиву.
- Primary CTA: `Запросить расчёт монолитных работ`
- Business Owner review: `Yes`
- Proof assets needed: минимум 1 сильный монолитный кейс, фото скрытых и промежуточных работ, scope по типам работ, фактология по технадзору / этапности, FAQ по срокам и подготовке участка.

### 3.3 Service 03

- Working title: `Строительство гостевых домов`
- Page intent: локально релевантный intent для малых гостиничных / гостевых объектов и частных инвестпроектов.
- Candidate slug: `stroitelstvo-gostevyh-domov`
- Working H1: `Строительство гостевых домов в Сочи`
- Role in launch-core: Sochi-specific page для hospitality-shaped спроса без ухода в большой dev/spec.
- Primary CTA: `Обсудить проект гостевого дома`
- Business Owner review: `Yes`
- Proof assets needed: минимум 1 объект, который реально подтверждает многокомнатный / гостевой сценарий, фото, планировочная фактура, аккуратные формулировки без инвестиционных обещаний, FAQ по срокам и этапам.

### 3.4 Service 04

- Working title: `Реконструкция и капитальный ремонт зданий`
- Page intent: запросы, где заказчик улучшает, перестраивает или капитально обновляет существующий объект.
- Candidate slug: `rekonstrukciya-kapitalnyy-remont-zdaniy`
- Working H1: `Реконструкция и капитальный ремонт зданий в Сочи`
- Role in launch-core: отдельный intent, не сводимый к новому строительству; полезен и для частных, и для небольших коммерческих объектов.
- Primary CTA: `Запросить выезд и оценку объекта`
- Business Owner review: `Yes`
- Proof assets needed: before/after кейс, описание исходного состояния и результата, границы ответственности, FAQ "когда реконструкция уместна", фото проблемных узлов и финала.

### 3.5 Service 05

- Working title: `Фасадные работы`
- Page intent: запросы на фасад, отделку, защиту внешнего контура и обновление внешнего вида здания.
- Candidate slug: `fasadnye-raboty`
- Working H1: `Фасадные работы в Сочи`
- Role in launch-core: узкий but practical service intent, который может конвертировать быстрее полного строительства.
- Primary CTA: `Получить расчёт фасадных работ`
- Business Owner review: `Yes`
- Proof assets needed: сильная фотофактура, материалы / виды работ, минимум 1 кейс с внятным результатом, FAQ по срокам и сезонности, осторожные формулировки по гарантии.

### 3.6 Service 06

- Working title: `Строительство коммерческих объектов`
- Page intent: B2B / owner-side intent на строительство нежилого объекта малого и среднего масштаба.
- Candidate slug: `stroitelstvo-kommercheskih-obektov`
- Working H1: `Строительство коммерческих объектов в Сочи`
- Role in launch-core: optional B2B slot; усиливает профиль компании, если proof inventory реально позволяет.
- Primary CTA: `Обсудить проект и получить предварительный scope`
- Business Owner review: `Yes`
- Proof assets needed: хотя бы 1 реальный нежилой / коммерческий кейс, понятные границы scope, SRO / подрядная фактология, аккуратный список типов объектов.

### Publish rule for Service 06

Если у команды нет убедительного коммерческого кейса, фото и фактологии, `Service 06` не выходит в launch and moves to phase `1.5`. Лучше выпустить `5` сильных money pages, чем `6` с одной слабой и недоказанной.

## 4. Launch Case-Page Core

Рекомендуемое ядро: `4` case pages. Названия ниже рабочие: их нужно привязать к реальному inventory, а не придумывать с нуля.

### 4.1 Case 01

- Working title: `Частный дом на участке со сложным рельефом`
- Confirms: что компания умеет строить в локальном сочинском контексте, где важны геология, уклон, дренаж, подпорные решения.
- Connected service pages: `Строительство домов под ключ`, `Монолитные работы`
- Missing materials to check: полный фотонабор этапов, план / площадь, сроки, цитата клиента, описание инженерных и рельефных ограничений.

### 4.2 Case 02

- Working title: `Монолитный каркас и конструктив частного объекта`
- Confirms: конструктивную компетенцию, hidden works discipline, техническую доказательность.
- Connected service pages: `Монолитные работы`, `Строительство домов под ключ`
- Missing materials to check: фото арматуры / опалубки / бетонирования, краткий scope работ, факты по срокам, что именно делала компания.

### 4.3 Case 03

- Working title: `Реконструкция существующего здания`
- Confirms: что компания умеет работать не только "с чистого листа", но и с существующим объектом.
- Connected service pages: `Реконструкция и капитальный ремонт зданий`, `Фасадные работы`
- Missing materials to check: before/after, исходная проблема, список работ, ограничения на объекте, итоговый результат в понятной форме.

### 4.4 Case 04

- Working title: `Гостевой или небольшой коммерческий объект`
- Confirms: hospitality / commercial competence для локального рынка.
- Connected service pages: `Строительство гостевых домов`, `Строительство коммерческих объектов`
- Missing materials to check: вообще есть ли релевантный объект; без него страницы про guest/commercial нельзя делать главными в launch-core.

## 5. Minimum Supporting Content

Нужен не "большой блог", а минимальный supporting contour, который:

- поддерживает money pages;
- снимает типовые возражения;
- помогает internal linking;
- даёт несколько индексируемых supporting URLs без распухания.

### 5.1 Supporting article topics

- `Как подготовить участок к строительству дома в Сочи: рельеф, геология, дренаж`
- `Что влияет на стоимость строительства дома в Сочи: не только квадратные метры`
- `Монолит, блок или комбинированная схема: что учитывать в сочинском климате`
- `Когда реконструкция выгоднее нового строительства`

### 5.2 FAQ clusters

- `Дом под ключ`: этапы, сроки, что входит, что готовит заказчик, как идёт контроль работ.
- `Монолит и конструктив`: фундамент, подпорные стены, скрытые работы, контроль качества.
- `Реконструкция / фасад`: когда нужен выезд, какие данные нужны для оценки, сезонность, ограничения.

### 5.3 Trust / support pages

- `About` должен содержать не только описание компании, но и proof blocks: реквизиты, SRO / разрешительные сигналы, процесс работы, ключевые роли, реальную фактуру по проектам.
- `Contacts` должен быть полноценной lead page: primary region cluster, способы связи, мессенджеры, форма, адрес / офисный якорь, service area statement без географического расползания.

### What not to add at launch

- не нужен большой "экспертный блог";
- не нужен отдельный калькулятор;
- не нужна отдельная AI page;
- не нужна индексируемая сетка фильтров или десятки статей "ради SEO".

## 7. Owner Review Map

### Mandatory Business Owner review

- `/`
- все launch money pages при первой публикации
- `/about`
- flagship cases
- любые страницы, где есть сильные обещания про цены, сроки, гарантии, инвестиционную окупаемость, "лучший / №1" claims или спорные формулировки

### Standard editorial flow is acceptable

- `/contacts`
- supporting articles
- FAQ updates
- routine proof-led updates на case pages, если они не меняют коммерческий смысл и не добавляют спорных обещаний

## 8. Assumptions and Gaps

- Точный service scope компании не подтверждён текущим inventory; этот spec даёт launch recommendation, а не финальную "истину по услугам".
- Рекомендация по региону основана на видимом офлайн-якоре компании и локальных SERP-сигналах, а не на полном keyword research.
- `Строительство гостевых домов` и особенно `Строительство коммерческих объектов` нельзя делать сильными launch pages без реального proof inventory.
- Нет подтверждённого списка кейсов, фотоархива, отзывов и верифицированной фактологии по каждому объекту.
- Неясно, какие именно обещания про сроки / стоимость / гарантии компания готова публично защищать.
- Если инвентаризация покажет слабую доказательную базу, ядро надо ужать до `5` service pages и `3` case pages.

## 9. Recommended Next Step After This Document

Следующий шаг после этого spec:

1. Подтвердить owner-level два решения: primary region cluster и финальный launch list money pages.
2. Провести content inventory + proof audit по каждому service page и case page.
3. Вычеркнуть всё, что не проходит по доказательной базе.
4. Превратить оставшееся ядро в editorial backlog: page brief, needed assets, owner review flag, publish readiness.

Правильный результат следующего шага: не "ещё один большой SEO-документ", а короткая, рабочая launch-очередь на производство контента.

## AI Discoverability — launch checklist

- [ ] HTML discovery hint comments добавлены в корневой layout публичного `next-app`
- [ ] `<meta name="ai-content-hints">` добавлен в корневой layout
- [ ] Статический стартовый маршрут `/llms.txt` подготовлен и отдается публично через текущий `next-app`
- [ ] Динамический public read-side endpoint `/llms-full.txt` реализован и возвращает только опубликованный контент
- [ ] Schema.org JSON-LD реализован для всех типов страниц согласно таблице в PRD
- [ ] `Global Settings` используется как единый source of truth для schema данных компании
- [ ] FAQ-блоки на service pages сформулированы как реальные пользовательские вопросы

# PAGES_SINGLE_WORKFLOW_Экостройконтинент_v1

Статус: active refinement doc
Дата: 2026-04-10
Назначение: каноническая UX-модель домена `Страницы` после отказа от модели двух соседних экранов `Страницы / AI-верстка`.

## 1. Product position

Пользовательский домен один: `Страницы`.

Unified `page workspace` остаётся одним canonical operator surface для домена `Страницы`. AI не становится отдельным продуктовым surface первого уровня и не перехватывает ownership истины. Он может появляться только как встроенная assistive-возможность внутри этого же экрана и не считается отдельным обязательным visual surface для каждой implementation wave.

Эта модель нужна, чтобы убрать:
- ownership drift;
- двойной вход в один page workflow;
- ощущение второго редактора страницы;
- engineer-dashboard posture на основном рабочем экране.

### Scope update for multi-type evolution

Этот документ нужно читать не только как канон для standalone static pages, но и как канон для multi-type page workspace.

Это означает:

- один editor surface остаётся правилом;
- расширение идёт через `page types`, а не через новые top-level editors;
- `about` и `contacts` остаются valid page types, но не исчерпывают весь домен;
- landing-like page types могут добавляться внутрь этого же workspace;
- source domains остаются first-class и не растворяются в page text;
- если для landing-а появляется второй competing editor, это считается нарушением этого канона.

## 2. First-level screen: реестр страниц

Экран `Страницы` - это обзорный вход в работу.

### Default posture

По умолчанию экран открывается в режиме карточек.

На экране доступны:
- переключатель `Карточки / Список`;
- поиск;
- фильтры;
- быстрый вход в создание новой страницы.

### Минимальная карточка страницы

Карточка страницы должна быть короткой и лёгкой. В первом слое видны только:
- название;
- превью;
- статус;
- меню `три точки`.

Карточка не должна притворяться вторым dashboard-пакетом и не должна вываливать SEO, технические поля, source refs или внутреннюю диагностику.

### Поведение карточки

- Клик по карточке открывает главный рабочий экран страницы.
- `Три точки` открывают компактное меню действий.
- Из меню действий доступен вход в метаданные страницы.

## 3. Metadata posture

Метаданные - это не главный экран и не основной способ мыслить о странице.

Метаданные живут в отдельном управленческом слое:
- открываются по `три точки` с карточки или изнутри страницы;
- оформлены как вкладочная модалка;
- модалка должна быть двигаемой, чтобы пользователь мог подглядывать контекст страницы.

### Роль metadata layer

В metadata layer попадают редкие, служебные и управленческие поля:
- route / slug / page type;
- SEO и preview metadata;
- publish-adjacent служебные настройки;
- поля, которые нужны не на каждом проходе по странице.

Принцип: частое и композиционное остаётся в основном page workflow; редкое и служебное уходит в metadata layer.

## 4. Main page workspace

У каждой страницы есть один главный рабочий экран.

Это единый page workspace, а не отдельный AI screen и не второй параллельный owner workflow.

### Основная компоновка

- центр = мольберт / canvas / story rail страницы;
- слева = компактная панель источников;
- справа = rail предпросмотра и готовности;
- assistive AI, если он surfaced в текущей wave, живёт внутри этого же workspace и не образует отдельный экран.

### UX posture

Этот экран должен помогать SEO-специалисту быстро собрать внятную страницу, а не читать инженерное состояние системы.

Поэтому:
- техническую диагностику нельзя поднимать в первый слой;
- редкие поля нельзя держать постоянно открытыми в центре;
- page workspace не должен ощущаться как ЦУП или prompt-lab.

## 5. Left source panel

Левая панель - это не склад длинных списков сущностей и не бесконечный скроллящийся каталог карточек.

В первом слое она показывает только компактные точки входа. Для первого среза:
- `Медиа`
- `Кейсы`
- `Услуги`

По клику открываются специализированные модалки / галереи выбора.

Сами launcher families могут эволюционировать по мере появления новых page types и новых source domains. Но правило остаётся тем же:

- left rail stays compact;
- source domains stay first-class;
- page workspace не превращается в scrolling warehouse.

### Source selection rule

Пользователь не должен просматривать длинный список всех сущностей прямо в левом rail. Он должен:
- выбрать тип источника;
- открыть удобную модалку выбора;
- взять нужный материал;
- вернуться в page canvas.

Это уменьшает визуальный шум и делает page workspace легче для повседневой SEO-работы.

## 6. Composition and connective text

Страница является владельцем композиции, переходов и связок между блоками.

Из этого следуют два правила:
- связочный текст не является внешней чужой сущностью;
- связочный текст не должен выпадать в отдельный detached workflow.

### Product rule

Connective copy - часть page composition workflow.

Он должен редактироваться там, где собирается история страницы:
- в контексте block flow;
- рядом с местом, где нужен переход;
- как page-owned composition text.

Нельзя описывать связочный текст как отдельную библиотечную сущность только ради UI-удобства. Это сломает ownership страницы.

## 7. AI posture

AI остаётся встроенной assistive-возможностью внутри page workspace.

AI может:
- помогать с draft copy;
- предлагать bridging text;
- предлагать усиление блока;
- помогать с формулировкой переходов и CTA.

AI не должен:
- выглядеть отдельным top-level screen;
- создавать второй owner workflow;
- перехватывать canonical truth страницы;
- превращать основной экран в инженерную панель состояний.

## 8. Interaction hierarchy

Главный экран должен быть лёгким и рабочим для SEO-специалиста.

### В первый слой выносим

- саму страницу и её композицию;
- понятные источники;
- понятные операторские подсказки и, если текущая wave это поддерживает, bounded AI-действия;
- быстрый handoff к метаданным по требованию.

### Во второй слой уводим

- редкие metadata-поля;
- служебные настройки;
- тяжёлые списки сущностей;
- техническую диагностику и trace-like детали.

## 9. Explicit non-goals

- Не делать отдельный top-level экран `AI-верстка`.
- Не делать из page workspace второй owner truth.
- Не превращать центр в свободный page builder canvas.
- Не вытаскивать служебные metadata-поля в основной первый слой без веской причины.
- Не делать левый rail бесконечной библиотекой карточек.
- Не превращать встроенную AI-помощь в prompt lab.

## 10. Practical documentation rule

Если соседний продуктовый или engineering документ описывает:
- `AI-верстка` как отдельный top-level surface;
- chooser как обязательный первый вход в работу со страницей;
- page workflow как два равноправных соседних сценария;
- landing workflow как отдельный competing editor рядом с `Страницы`;
- engineer-dashboard posture как целевой UX,

то такой документ должен считаться устаревшим и подлежит выравниванию под этот single-workflow model.

## 11. Test Data Naming Rule

Для page/workspace flows действует отдельный operational canon:

- любой временный объект, который агент или разработчик создаёт для smoke / audit / remediation / archive verification / preview check, должен быть явно test-marked;
- обязательный минимальный инвариант: имя начинается с `test__`;
- рекомендуемый формат: `test__<domain>__<purpose>__<suffix>`.

Примеры:

- `test__page__smoke`
- `test__page__archive-fixture__20260411`
- `test__service__picker-fixture__01`

Это правило нужно не ради косметики, а чтобы:

- не смешивать реальную ценность и disposable инженерный мусор;
- удешевлять будущий cleanup;
- делать test artifacts легко заметными для следующего агента даже в empty-context чате.

Отдельная осторожность нужна для `media domain`:

- реальных media в проекте мало и они ценны;
- test media нельзя создавать без крайней необходимости;
- если test media всё же создаётся, он тоже обязан быть test-marked.

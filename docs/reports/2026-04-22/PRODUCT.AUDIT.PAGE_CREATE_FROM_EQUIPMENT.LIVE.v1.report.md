# PRODUCT.AUDIT.PAGE_CREATE_FROM_EQUIPMENT.LIVE.v1.report

Дата: 2026-04-22
Среда проверки: live domain `https://ecostroycontinent.ru`
Режим: продуктовый аудит, без рефакторинга

## 1. Цель

Зафиксировать фактическое поведение сценария создания страницы из техники в админке и собрать замечания как входные данные для следующего рефакторинга.

## 2. Короткий вывод

Сценарий `Страницы -> Новая страница -> Из техники` в текущем live-состоянии не является рабочим операторским потоком.

Проблема не только в том, что в модалке доступен выбор только одной единицы техники. Выше по приоритету то, что сам сценарий после submit блокируется серверной валидацией и не доводит оператора до полноценного редактора страницы.

## 3. Что было проверено

1. Открыт live-домен `https://ecostroycontinent.ru/admin`.
2. Открыт раздел `Страницы`.
3. Вызвана registry-native модалка `Новая страница`.
4. Выбран режим старта `Из техники`.
5. Проверено поле источника техники.
6. Выполнена попытка создать тестовую страницу с тестовым заголовком.
7. Дополнительно открыт canonical fallback route `/admin/entities/page/new`.
8. Выполнен контрольный сценарий создания `Отдельная страница`, чтобы отделить проблему equipment-flow от общей поломки create-flow.

## 4. Основные findings

### F-001. Сценарий `Из техники` открыт в UI, но фактически заблокирован

Severity: critical

Факт:
- Модалка позволяет выбрать режим старта `Из техники` как нормальный пользовательский сценарий.
- После нажатия `Создать и открыть` live-система возвращает ошибку:
  `Launch ownership guard blocks legacy pageType "equipment_landing". Allowed standalone page types: about, contacts.`

Следствие:
- UI обещает рабочий путь создания страницы из техники, но фактически этот путь недоступен.
- Для оператора это выглядит как broken flow, а не как осознанное ограничение продукта.

### F-002. Источник техники в модалке поддерживает только одну технику

Severity: high

Факт:
- В режиме `Из техники` показан один обычный select `primaryEquipmentId`.
- DOM-инспекция на live-странице подтвердила: `multiple=false`.

Следствие:
- На стартовом шаге оператор может выбрать только одну технику.
- Если продуктовая модель предполагает composite-страницу по нескольким единицам техники, текущая форма этому противоречит.

### F-003. Ограничение `одна техника` не компенсируется в полном редакторе

Severity: high

Факт:
- На canonical route `/admin/entities/page/new` в полном редакторе страницы в блоке `Связи` есть только:
  - связанные услуги;
  - связанные кейсы;
  - коллекции.
- Отдельной поверхности для множественного выбора техники в редакторе не найдено.

Следствие:
- Нельзя сказать, что create-modal выбирает одну primary-технику, а потом оператор расширяет набор в workspace.
- По факту equipment relation сейчас выглядит как single-source модель без второго шага для multi-select.

### F-004. Продуктовая коммуникация экрана противоречива

Severity: medium

Факт:
- В фильтре типов страниц на реестре видно `Страница техники (устаревшая, не для запуска)`.
- Одновременно в primary create-modal режим `Из техники` всё ещё присутствует как активный сценарий создания.

Следствие:
- Оператору неясно, это поддерживаемый путь или legacy-остаток.
- Наличие действия в primary create entry воспринимается как разрешённый сценарий.

### F-005. Модалка не является финальной поверхностью по текущему канону

Severity: info

Факт:
- Registry-modal собирает стартовый контекст.
- При успешном создании отдельной страницы система переводит оператора в полноценный workspace `/admin/entities/page/[pageId]`.
- Контрольный сценарий `Отдельная страница` это подтвердил.

Следствие:
- Текущая модель продукта: модалка = create entry, полноценный editor = следующая поверхность.
- Проблема equipment-flow не в том, что редактора нет, а в том, что flow до него не доходит.

### F-006. Fallback link внутри модалки не выглядит надёжной точкой спасения сценария

Severity: medium

Факт:
- В модалке присутствует ссылка `Полный fallback-маршрут`.
- Во время live-прогона элемент существовал, но оказался неудобен для взаимодействия из-за положения вне viewport.

Следствие:
- Даже аварийный путь не выглядит достаточно надёжным и очевидным.
- Для оператора это не компенсирует broken primary flow.

## 5. Контрольное сравнение

### Что работает

Сценарий `Отдельная страница` успешно создал страницу и открыл полноценный редактор.

Факт открытия:
- redirect произошёл на `/admin/entities/page/entity_75695b3b-9f93-454d-8173-774a192fe823?message=Черновик+сохранён.`

Вывод:
- create-flow в целом не сломан;
- проблема локализуется именно в ветке `Из техники`.

## 6. Продуктовые вопросы для следующего рефакторинга

1. Является ли `Из техники` поддерживаемым create-scenario, или это legacy-режим, который должен быть убран из primary create-modal?
2. Если сценарий поддерживается, какая продуктовая модель верна:
   - одна primary техника;
   - несколько техник;
   - одна primary + дополнительные связанные техники?
3. Где оператор должен собирать equipment set:
   - прямо в create-modal;
   - в полном workspace после создания;
   - в отдельном источниковом блоке внутри workspace?
4. Должен ли launch-ownership guard блокировать этот flow целиком, или UI должен заранее не предлагать сценарий, который guard не пропускает?

## 9. Addendum: live audit карточки `Техника`

Дата: 2026-04-22
Среда проверки: live domain `https://ecostroycontinent.ru`
Проверенный экран:
- `/admin/entities/equipment`
- `/admin/entities/equipment/entity_85793a87-4d33-4f3b-9da2-0af0ba8ee612?returnTo=%2Fadmin%2Fentities%2Fequipment`

### 9.1. Короткий вывод

Карточка техники в текущем live-виде выглядит визуально тяжёлой и неэкономно расходует экранное пространство.

По факту данных на экране не так много, но они упакованы в слишком высокий верхний hero-блок, крупные action-pill элементы и массивную правую диагностическую колонку. Из-за этого первый экран занят mostly chrome и статусами, а до плотной полезной формы оператор доходит только после заметного скролла.

### 9.2. Что подтвердилось в live-проверке

#### E-001. Верхняя hero-зона слишком высокая для реального объёма информации

Severity: high

Факт:
- При viewport `1440x900` первый содержательный блок редактирования `Что хотим изменить` начинается только на высоте `top=777`.
- До основного edit-flow экран в основном занят:
  - breadcrumb + title;
  - большой блок `Рабочая карточка`;
  - отдельный ряд действий под ним;
  - правый diagnostic rail.

Следствие:
- Первый экран уходит не на редактирование, а на крупную презентационную шапку.
- Даже до начала основной формы оператор доходит почти у нижней границы viewport.

#### E-002. Экран чрезмерно длинный для такого набора полей

Severity: high

Факт:
- При viewport `1440x900` общая высота страницы составила `6192px`.
- Кнопка `Сохранить черновик` находится на `top=6089`, то есть почти через семь экранов прокрутки.

Следствие:
- Для карточки с умеренным объёмом данных это выглядит как layout inflation, а не как реальная содержательная сложность.
- Возникает ощущение громоздкого экрана даже при относительно небольшом доменном payload.

#### E-003. Правая колонка съедает заметную площадь, но почти не помогает редактированию

Severity: high

Факт:
- Справа постоянно занимают место крупные блоки:
  - `Проверка готовности`;
  - `Реестр доказательств`;
  - `Опубликованная версия`;
  - `Лента аудита`.
- В observed state эта колонка mostly diagnostic/read-only.
- В `Реестр доказательств` прямо написано, что редактируемые элементы в панели отсутствуют.

Следствие:
- Правая колонка визуально тяжёлая, но не даёт сопоставимой operational value на первом экране.
- Для SEO-оператора или контент-редактора она выглядит как постоянный инженерный ballast рядом с основной формой.

#### E-004. Есть дубли действий и action clutter в верхней зоне

Severity: medium

Факт:
- `История` присутствует как минимум трижды:
  - внутри hero summary;
  - в ряду top actions;
  - снова внизу рядом с `Сохранить черновик` / `Отправить на проверку`.
- В верхней action zone рядом собраны:
  - `История`;
  - `Пометить на удаление`;
  - `Центр очистки`;
  - `Снять с публикации`;
  - `Безопасно убрать (legacy)`.

Следствие:
- Визуальный приоритет действий размыт.
- Второстепенные и legacy-операции спорят за внимание с основным редактированием.

#### E-005. Лейбл `Поисковая оптимизация / данные` повторяется слишком много раз

Severity: high

Факт:
- В live-DOM этот eyebrow повторяется `6` раз на одном экране.
- Он предваряет почти каждую секцию, включая те, которые по смыслу не являются SEO-first surface.

Следствие:
- Это подтверждает ощущение дублирования и перегруза терминологией.
- Вместо ясной структуры секций оператор видит повторяющийся инженерный ярлык.

#### E-006. Терминологический слой остаётся слишком инженерным для SEO-оператора

Severity: medium

Факт:
- На верхнем экране рядом с обычным редактированием присутствуют:
  - `Реестр доказательств`;
  - `Центр очистки`;
  - `Безопасно убрать (legacy)`;
  - объяснения про truth, домены и доказательный слой.

Следствие:
- Экран выглядит как смесь рабочего редактора и инженерной/системной панели.
- Это усиливает ощущение тяжёлой админки вместо компактной контентной карточки.

#### E-007. Крупные pills, кнопки и отступы визуально раздувают экран

Severity: medium

Факт:
- Статусы, версии, предупреждения и действия оформлены крупными capsule/pill элементами.
- Между hero, action-row и секциями формы большие вертикальные интервалы.
- Даже компактные по смыслу данные занимают отдельные большие карточки.

Следствие:
- Плотность полезной информации низкая.
- Оператор получает много крупного UI chrome и мало “рабочей концентрации” на первом экране.

### 9.3. Продуктовая интерпретация замечаний

Если держаться исходной идеи домена, где:
- медиа;
- услуги;
- кейсы;
- техника

сначала существуют как source entities, а уже потом становятся источниками для сборки page/landing surface, то карточка техники должна быть в первую очередь compact source-of-truth editor, а не большой hybrid-dashboard.

По observed live-поведению сейчас экран техники ощущается скорее как:
- heavy admin surface;
- с большим количеством системной обвязки;
- с низкой информационной плотностью;
- и с заметным приоритетом diagnostic chrome над самой domain editing task.

### 9.4. Что стоит проверить в следующем рефакторинге

1. Можно ли убрать hero inflation и сократить высоту верхнего summary-блока.
2. Можно ли сделать right rail collapseable, secondary или context-driven, а не permanently dominant.
3. Какие верхние действия реально нужны на первом экране, а какие лучше увести в menu / secondary affordance.
4. Нужно ли оставить repeated eyebrow `Поисковая оптимизация / данные` у каждой секции, или структура должна стать короче и человечнее.
5. Можно ли довести экран техники до one-screen-first editing posture на desktop без заметного initial scroll.

## 10. Cross-screen refactor note: это похоже на системную проблему, а не только на экран `Техника`

Дата фиксации гипотезы: 2026-04-22
Источник:
- live-аудит flow создания страницы;
- live-аудит карточки `Техника`;
- product feedback от владельца продукта по другим admin screens.

### 10.1. Короткий вывод

На текущем этапе это уже выглядит не как isolated defect одной сущности, а как повторяющийся UX-pattern административного интерфейса.

Рабочая гипотеза высокого доверия:
- большинство entity screens в админке унаследовали один и тот же layout language;
- этот layout language системно завышает визуальный вес chrome-слоя;
- из-за этого экраны получаются крупнее, тяжелее и длиннее, чем требует реальный объём данных.

### 10.2. Повторяющиеся признаки системной проблемы

1. Слишком высокий верхний экран до начала основной формы.
2. Крупные hero-блоки и summary-панели при умеренном объёме доменных данных.
3. Правые колонки, которые занимают площадь постоянно, хотя полезны лишь контекстно.
4. Большие pills, статусы, предупреждения и action rows, раздувающие вертикальный ритм.
5. Повторяющиеся секционные ярлыки и терминологические дубли вместо компактной структуры.
6. Смешение рабочего редактора с инженерной/диагностической поверхностью.
7. Низкая плотность полезной информации на первом экране.

### 10.3. Почему это важно для рефакторинга

Если воспринимать каждый экран как отдельную локальную проблему, рефакторинг рискует превратиться в серию частных cosmetic fixes:
- чуть уменьшить hero на `Технике`;
- чуть спрятать один блок на `Услугах`;
- чуть поджать кнопки на `Кейсах`.

Но по текущим наблюдениям корень проблемы, вероятно, общий:
- переусложнённый admin shell;
- завышенный scale базовых UI primitives;
- слабое разделение primary editing surface и secondary diagnostic surface;
- избыточное присутствие инженерного языка в экранах, которыми должен пользоваться предметный специалист.

### 10.4. Предварительная продуктовая формулировка для общего рефакторинга

Цель следующего этапа стоит формулировать не как “подправить экран техники”, а как:

`Сделать административные карточки сущностей компактными, лёгкими и редактирующими-first экранами, где основная доменная задача помещается в первый экран, а системные и диагностические блоки становятся вторичными, контекстными и менее шумными.`

### 10.5. Что теперь проверять уже как общую программу аудита

1. Какие admin screens реально помещают основной edit-flow в первый desktop viewport, а какие нет.
2. Где right rail обязателен, а где он должен стать сворачиваемым или быть вынесен ниже.
3. Какие action patterns повторяются между сущностями и могут быть унифицированы в более компактный toolbar.
4. Какие лейблы и секционные eyebrow-элементы дублируются по всей админке.
5. Где инженерная терминология просачивается в surface, который должен быть понятен SEO-специалисту или контент-редактору.
6. Можно ли выработать единый density standard для всех entity cards: заголовок, actions, sidebar, секции формы, sticky-save area.

### 10.6. Статус утверждения

Важно:
- это уже не единичное замечание по `Технике`;
- но это ещё не завершённый полный аудит всех сущностей.

То есть на сегодня корректно фиксировать вывод так:
- `Техника` уже дала сильное подтверждение паттерна;
- flow `Страница из техники` показал ту же общую тяжесть и перегруженность;
- владелец продукта отдельно подтверждает, что аналогичное ощущение есть практически на всех экранах;
- следующий проход нужно делать как systematic cross-screen audit, а не как разрозненные точечные правки.
